# 프로세스 vs 스레드

## 한 줄 정의

- **프로세스(Process)**: 실행 중인 프로그램 자체
- **스레드(Thread)**: 프로세스 내부에서 실제 작업을 수행하는 실행 흐름

## 핵심 차이 한 줄 요약

프로세스는 메모리를 독립적으로 사용하고, 스레드는 메모리를 공유한다.

## 비유로 이해하기

### 프로세스 = 집 🏠

프로세스는 각각 독립된 집이다.

- 옆집이 불나도 우리 집은 안전
- 다른 프로세스 메모리에 직접 접근 불가능
- 안정성이 높음

프로세스는 운영체제로부터 메모리, CPU 자원, 파일, 네트워크 자원 등을 할당받는다.
즉, 프로세스는 **"자원 할당의 단위"**.

### 스레드 = 집 안 사람 👨‍👩‍👧‍👦

스레드는 같은 집 안에서 살아가는 사람이다.

- 거실, 주방, 화장실 같은 공용 공간을 함께 사용
- 하지만 각자의 방(Stack)은 따로 사용

즉, 스레드는 **"실행 흐름의 단위"**.

## 프로세스 메모리 구조

```
[ Code ]
[ Data ]
[ Heap ]
[ Stack ]
```

### Code 영역

실행할 프로그램 코드 저장 영역.

```java
public void hello() {
    System.out.println("hello");
}
```

- 읽기 전용
- 실행 코드 저장

### Data 영역

전역 변수(static 변수 포함) 저장 영역.

```java
static int count = 0;
```

- 프로그램 종료까지 유지
- 전역 데이터 저장

### Heap 영역

동적으로 생성된 객체 저장 영역.

```java
User user = new User();
```

- 런타임 중 메모리 할당
- Garbage Collection 대상

### Stack 영역

함수 호출 정보 및 지역 변수 저장 영역.

```java
int age = 10;
```

- 함수 호출 시 stack frame 생성
- 함수 종료 시 자동 제거
- 스레드마다 독립적으로 존재

## 스레드 메모리 구조

스레드는 `Code`, `Data`, `Heap`을 공유하고 `Stack`만 개별로 가진다.

```
공유:  Code / Data / Heap
개별:  Thread A Stack | Thread B Stack | Thread C Stack
```

- 공용 자원은 함께 사용
- 실행 흐름은 독립적

## 왜 스레드가 더 빠를까?

핵심은 **Context Switching 비용 차이**.

### Context Switching 이란?

CPU가 실행 대상을 바꾸는 과정. (Process A → Process B, 또는 Thread A → Thread B)

### 프로세스 Context Switching

- 메모리 주소 공간 변경
- CPU 캐시 초기화
- PCB(Process Control Block) 변경
- CPU 레지스터 교체

→ "집 자체를 이동하는 수준". 비용이 크고 무겁다.

### 스레드 Context Switching

- 같은 메모리 공간 사용
- 주소 공간 유지
- 캐시 활용 가능

→ "방에서 거실로 이동하는 수준". 훨씬 가볍고 빠르다.

## 멀티 스레드의 장점

- 빠른 처리 속도
- 메모리 절약
- 자원 효율 증가
- 응답 속도 향상

예시: 웹 서버, 게임 엔진, 채팅 서버, 영상 처리

## 동시성 문제 (스레드의 위험)

메모리를 공유하기 때문에 여러 스레드가 동시에 같은 데이터에 접근 가능 → 동시성 문제 발생.

### Race Condition

여러 스레드가 동시에 같은 데이터를 수정하는 문제.

```java
count++;
```

```
Thread A : count 읽음 → 1
Thread B : count 읽음 → 1
Thread A : 2 저장
Thread B : 2 저장
```

기대값은 `3`이지만 실제 결과는 `2`.

### Deadlock (교착 상태)

서로 자원을 기다리며 무한 대기하는 상태.

```
Thread A: Lock1 점유 → Lock2 대기
Thread B: Lock2 점유 → Lock1 대기
```

→ 둘 다 영원히 멈춤.

### Deadlock 해결 방법

1. **Lock 순서 통일** — 항상 같은 순서로 lock 획득 (예: 항상 Lock1 → Lock2)
2. **synchronized 범위 최소화** — 불필요한 lock 줄이기
3. **Timeout 사용** — 일정 시간 이상 대기 시 lock 포기
4. **Concurrent 자료구조 사용** — `ConcurrentHashMap`, `CopyOnWriteArrayList`

## 멀티 프로세스 vs 멀티 스레드

| 항목 | 멀티 프로세스 | 멀티 스레드 |
| --- | --- | --- |
| 메모리 | 독립 | 공유 |
| 안정성 | 높음 | 낮음 |
| 속도 | 상대적으로 느림 | 빠름 |
| Context Switching | 무거움 | 가벼움 |
| 데이터 공유 | IPC 필요 | 쉬움 |
| 구현 난이도 | 비교적 쉬움 | 상대적으로 어려움 |

## 언제 무엇을 사용할까?

### 멀티 프로세스 — 안정성이 중요한 경우

예: 크롬 브라우저 탭, Docker Container, 독립 서비스 구조
장점: 하나 죽어도 전체 영향 적음 / 단점: 무겁다

### 멀티 스레드 — 성능이 중요한 경우

예: 웹 서버 요청 처리, 게임 서버, 실시간 채팅, 비동기 작업 처리
장점: 빠르고 가볍다 / 단점: 동기화 문제 발생 가능

## 크롬 탭은 프로세스일까?

크롬은 **탭별 멀티 프로세스 구조**를 사용한다.

- 한 탭이 죽어도 전체 브라우저가 죽지 않게
- 보안 격리
- 안정성 향상

→ 안정성을 위해 멀티 프로세스를 선택한 구조.

## 핵심 면접 답변 정리

프로세스는 운영체제로부터 자원을 할당받는 독립 실행 단위이고, 스레드는 프로세스 내부에서 실제 작업을 수행하는 실행 흐름입니다. 프로세스는 메모리를 독립적으로 사용하지만, 스레드는 Code, Data, Heap 영역을 공유하고 Stack만 독립적으로 사용합니다. 따라서 스레드는 Context Switching 비용이 적어 더 가볍고 빠릅니다.
