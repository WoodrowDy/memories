# 하네스 엔지니어링 (Harness Engineering)

> LLM 자체가 아니라 **LLM 주변을 설계하는 일**. 같은 모델로 더 빠르게, 더 안정적으로, 더 예측 가능하게 작동하도록 환경을 짜는 기술.

---

## 1. 하네스 엔지니어링이란?

### 모델 vs 하네스
- **모델**: Claude, GPT 같은 추론 엔진. 일반적으로 우리가 직접 만들지 않음.
- **하네스**: 모델이 작동하는 환경 — hooks, slash commands, MCP servers, permissions, skills, settings.json.

좋은 하네스는 같은 모델이 **반복 작업을 자동화**하고, **잊어버리는 단계를 강제**하고, **컨텍스트를 절약**하게 한다.

### 하네스 구성 요소

| 레이어 | 역할 | 예시 |
|---|---|---|
| **Skills** | 재사용 가능한 워크플로 | `/ship`, `/qa-only`, `/office-hours` |
| **Slash commands** | 단축 진입점 | `/review`, `/codex` |
| **Hooks** | 이벤트 자동화 | `SessionStart`, `UserPromptSubmit`, `PreToolUse` |
| **MCP servers** | 외부 도구 연결 | Gmail, Drive, GitHub, 커스텀 서버 |
| **Permissions** | 자동 승인 도구 화이트리스트 | `Bash(npm:*)`, `Read(**)` |
| **Settings** | 환경/모델/키바인딩 | `settings.json`, `keybindings.json` |
| **Subagents** | 격리된 컨텍스트 작업자 | `Explore`, `Plan`, 커스텀 에이전트 |

### 왜 중요한가

1. **반복 제거** — 매번 "테스트 돌리고, 린트 돌리고, PR 만들고…" 설명 안 해도 됨
2. **품질 게이트 내장** — 잊기 쉬운 단계(보안 검토, 디자인 감사, 회고)를 워크플로에 박음
3. **컨텍스트 절약** — 스킬이 프로젝트 컨벤션을 알고 있음 (CLAUDE.md 같은)
4. **공유 가능** — 베스트 프랙티스를 npm 패키지처럼 묶어서 팀에 배포

### 하네스 엔지니어링의 핵심 원칙

- **에비던스 > 가정** — 동작을 측정하고 검증할 것 (gstack은 LLM-judge eval로 회귀 감지)
- **Layer 3 사고** — 런타임/프레임워크에 빌트인이 있으면 그걸 써라. 직접 만들지 마라
- **Search before building** — 같은 문제를 푼 사람이 있다 (Anthropic skill-creator, OpenAI skills 등)
- **Boil the lake** — 절반만 만든 워크플로는 안 만든 것보다 못함

## 2. 지스택처럼 스킬 만드는 법

### 단순 스킬 vs 지스택식 스킬

**단순 스킬:** `~/.claude/skills/<name>/SKILL.md` 파일 하나. 손으로 작성. 빠르게 만들 때 OK.

**지스택식 스킬:** 템플릿 → 제너레이터 → 검증 → 설치 파이프라인.

지스택 방식의 추가 가치:
- 공통 블록(preamble, browse setup, gbrain context) 재사용
- 토큰 한도 자동 검사 (100KB 미만)
- 무료 검증 (`bun test`) + 유료 LLM-as-judge eval
- 멀티 호스트 지원 (Claude Code, Codex, Factory, OpenCode, Cursor 등)
- 글로벌/프로젝트 설치를 `./setup` 한 번으로

### 스킬 디렉토리 구조

```
my-skill/
├── SKILL.md.tmpl    # 템플릿 (편집 대상)
└── SKILL.md         # 자동 생성됨 (커밋은 함, 직접 편집 금지)
```

### SKILL.md.tmpl 헤더 포맷

```yaml
---
name: my-skill
preamble-tier: 3              # 1=최소, 2=보통, 3=풀 preamble
version: 1.0.0
description: |
  스킬이 뭘 하는지 한 문장.
  언제 호출되는지 (트리거 문장 예시).
  자동 호출 조건 (사용자가 X라고 말할 때 proactively 호출).
allowed-tools:
  - Bash
  - Read
  - Write
  - AskUserQuestion
triggers:
  - my-skill
  - 이 스킬 호출하는 자연어 키워드
---

{{PREAMBLE}}        # 공통 preamble 자동 주입 (verification, AskUserQuestion 형식 등)
{{BROWSE_SETUP}}    # 브라우저 자동화 필요할 때만

# My Skill

당신은 ___입니다. 당신의 역할은 ___입니다.

## Phase 1: ___
...
```

### 단계별 체크리스트

**0. 비슷한 게 이미 있는지 확인**
```bash
ls /Users/rowdy/Desktop/workspace/.claude/skills/gstack/
```
유사 스킬을 베이스로 복사하면 빠름.

**1. 디렉토리 + 템플릿 생성**
```bash
cd /Users/rowdy/Desktop/workspace/.claude/skills/gstack
mkdir my-skill
cp office-hours/SKILL.md.tmpl my-skill/SKILL.md.tmpl
# 헤더와 본문 편집
```

**2. (선택) 새 공통 블록 필요하면 resolver 추가**
- `scripts/resolvers/` 에 새 파일 (예: `my-block.ts`)
- `scripts/gen-skill-docs.ts` 에서 import 후 `{{MY_BLOCK}}` 토큰 등록
- 다른 스킬에서도 `{{MY_BLOCK}}` 한 줄로 재사용 가능

**3. SKILL.md 생성**
```bash
bun run gen:skill-docs
```
또는 watch 모드로 자동 재생성:
```bash
bun run dev:skill
```

**4. 검증**
```bash
bun test               # 무료 검증 (<2s) — 구문, 토큰 한도, gen-skill-docs 품질
bun run skill:check    # 모든 스킬 헬스 대시보드
```

LLM-judge eval이 필요하면 (~$0.15/run):
```bash
bun run test:evals
```

**5. 설치 (글로벌/프로젝트 등록)**
```bash
./setup
```
이게 top-level 심볼릭 링크를 만들어서 `/my-skill` 자동완성 가능하게 만듦.
설치 안 하면 `gstack/` 안에 중첩되어 Claude Code가 top-level 스킬로 인식 못 함.

**6. (선택) 다른 호스트에도 배포**
`./setup` 은 기본이 Claude. 다른 AI 에이전트에도 쓰려면:
```bash
./setup --host codex      # Codex CLI
./setup --host opencode   # OpenCode
./setup --host cursor     # Cursor
```

### SKILL.md.tmpl 작성 원칙

CLAUDE.md에서 발췌한 핵심 규칙:

- **자연어로 로직 표현** — SKILL.md.tmpl은 Claude가 읽는 **프롬프트 템플릿**이지 배쉬 스크립트가 아님. 각 bash 코드 블록은 별개 셸이라 변수가 안 넘어감. 대신 산문으로 "Step 0에서 감지한 base branch를 기억하라" 식으로 표현.
- **브랜치 이름 하드코딩 금지** — `main`/`master` 가정하지 말고 `gh pr view` 등으로 동적 감지. PR 타게팅에는 `{{BASE_BRANCH_DETECT}}` 토큰 사용.
- **bash 블록은 자기 완결적** — 이전 컨텍스트가 필요하면 산문으로 다시 명시.
- **조건문은 영어/한국어로** — 중첩된 if/elif 대신 번호 매긴 결정 단계: "1. X면 Y. 2. 아니면 Z."
- **플랫폼 종속 명령어 박지 마라** — 테스트/배포 명령은 CLAUDE.md에서 읽거나 AskUserQuestion으로 물어봐서 CLAUDE.md에 저장.

### 자주 막히는 곳

| 증상 | 원인 | 해결 |
|---|---|---|
| `/my-skill` 자동완성 안 됨 | top-level 심볼릭 링크 없음 | `./setup` 실행 |
| 토큰 한도 초과 경고 | SKILL.md > 100KB | 긴 섹션을 resolver로 분리, 평가 루브릭 압축 |
| 스킬이 호출 안 됨 | `triggers` 키워드/`description` 약함 | `skill-creator` 스킬로 description 최적화 |
| 머지 충돌 (SKILL.md) | 생성된 파일이 양쪽 다름 | `.tmpl`과 `gen-skill-docs.ts`만 충돌 해결 후 `bun run gen:skill-docs` 재실행 |
| 스킬이 다른 세션도 깸 | 심볼릭 링크 라이브 | 큰 리팩터링 중에는 `rm .claude/skills/gstack` 후 글로벌 설치 사용 |

### 본인이 만든 스킬을 지스택처럼 배포하기

지스택은 한 GitHub repo + `./setup` 스크립트로 작동. 같은 패턴 따라 하려면:

1. **Repo 만들기** — `~/your-skills/` 같은 곳에 git repo
2. **공통 인프라 복사** — `scripts/gen-skill-docs.ts`, `scripts/resolvers/preamble.ts`, `scripts/host-config.ts`, `setup` 스크립트를 베이스로
3. **호스트 설정 추가** — `hosts/claude.ts` 같은 typed config (멀티 호스트 지원하려면)
4. **CI 추가** — `.github/workflows/skill-docs.yml` 같은 걸로 SKILL.md 동기화 검증
5. **README + 설치 명령** — 사용자가 `git clone && ./setup` 하면 끝나도록

## 3. 더 깊이 파고들기

### 지스택에서 봐야 할 파일들

| 파일 | 보는 이유 |
|---|---|
| `SKILL.md.tmpl` | 메인 진입점 — 모든 다른 스킬의 디스패처 |
| `scripts/gen-skill-docs.ts` | 제너레이터 본체 — 템플릿 → SKILL.md 변환 로직 |
| `scripts/resolvers/preamble.ts` | 모든 스킬에 주입되는 공통 행동 규칙 |
| `scripts/host-config.ts` | 멀티 호스트 지원 인터페이스 |
| `test/skill-validation.test.ts` | 자동 검증 항목 |
| `test/helpers/llm-judge.ts` | LLM-as-judge eval 패턴 |
| `office-hours/SKILL.md.tmpl` | 잘 짜인 스킬 예시 (자연어 + Phase 구조) |
| `ship/SKILL.md.tmpl` | 13단계 워크플로 + 게이트 패턴 예시 |

### Anthropic 공식 도구
- **skill-creator** (현재 사용 가능) — 스킬 만들고 description 최적화하고 eval 돌리기
- **using-superpowers** — 스킬을 어떻게/언제 호출하는지 메타 가이드
- **writing-skills** — 스킬 작성 베스트 프랙티스

### 외부 레퍼런스
- Anthropic Skills 문서: https://code.claude.com/docs/ko/skills
- Claude Code hooks: settings.json `hooks` 섹션 — `update-config` 스킬 참고
- ETHOS.md (지스택) — "Boil the Lake", "Search Before Building" 빌더 철학

## 핵심 요약

> **하네스 엔지니어링 = LLM의 환경을 짜는 일**.
> 좋은 하네스는 모델 성능을 안 바꿔도 결과를 10배 좋게 만든다.
>
> **지스택식 스킬 = 단일 SKILL.md + 템플릿 파이프라인**.
> 제너레이터, 검증, 설치, 멀티 호스트 지원이 들어 있어서 한 번 짜면 오래 간다.
>
> **시작하는 법**: 비슷한 스킬 복사 → 헤더 편집 → `bun run gen:skill-docs` → `bun test` → `./setup`.
