# 데드락(Deadlock) vs 라이브락(Livelock)

> 백엔드(NestJS / Java Spring) 관점으로 정리한 동시성 교착 상태 비교 문서

---

## 1. 한 줄 정의

- **데드락(Deadlock, 교착 상태)**: 프로세스/스레드들이 자원을 **점유한 채로** 서로를 기다리며 **대기 상태(blocked)로 멈춰** 있는 상태.
- **라이브락(Livelock)**: 프로세스/스레드들이 서로 양보하느라 상태를 계속 바꾸며 **실행 중(running)이지만 작업은 진행되지 않는** 상태.

핵심 차이: **데드락은 서로 잡고 멈추고, 라이브락은 서로 양보하며 헛돈다.**

---

## 2. 핵심 비교표

| 구분 | 데드락 | 라이브락 |
|------|--------|----------|
| 상태 | blocked / waiting | running |
| CPU 사용량 | 거의 없음 (멈춰 있음) | 높음 (무한 재시도) |
| 자원 처리 | 잡고 안 놓음 | 놓았다 다시 잡기 반복 |
| 탐지 난이도 | 상대적으로 쉬움 (멈춰 있음) | 어려움 (일하는 것처럼 보임) |
| 진행 여부 | 둘 다 멈춤 | 둘 다 바쁘지만 진전 없음 |

---

## 3. 직관적 비유

**복도에서 두 사람이 마주쳤을 때**

- **데드락**: 둘 다 가만히 서서 "네가 먼저 가" 하며 아무도 안 움직임 → 멈춤(blocked)
- **라이브락**: 둘 다 오른쪽으로 비킴 → 또 마주침 → 둘 다 왼쪽으로 비킴 → 또 마주침 → 무한 반복. 서로 배려해서 계속 움직이지만(running, CPU 사용) 아무도 못 지나감.

---

## 4. 락 시나리오 예시

### 데드락 — 자원 순서를 거꾸로 잡음

계좌 이체를 두 트랜잭션이 동시에 **반대 방향**으로 처리하는 상황 (가장 흔한 교과서 사례).

```
[Transaction 1] A → B 이체              [Transaction 2] B → A 이체

T1: UPDATE accounts SET bal = bal-100
    WHERE id = 'A';   -- A행 락 획득 ✅
                                        T2: UPDATE accounts SET bal = bal-50
                                            WHERE id = 'B';   -- B행 락 획득 ✅

T1: UPDATE accounts SET bal = bal+100
    WHERE id = 'B';   -- B행 락 대기 ⏳
    (T2가 B를 쥐고 있어서 멈춤)
                                        T2: UPDATE accounts SET bal = bal+50
                                            WHERE id = 'A';   -- A행 락 대기 ⏳
                                            (T1이 A를 쥐고 있어서 멈춤)

        ⬇                                       ⬇
   T1은 B를 기다리고, T2는 A를 기다림 → 순환 대기 → 데드락
```

DB가 개입한다.

```
ERROR 1213 (40001): Deadlock found when trying to get lock;
try restarting transaction
```

MySQL은 둘 중 하나(예: T2)를 victim으로 골라 자동 롤백하고, T1은 정상 진행된다.
보통 **롤백 비용이 가장 적은 트랜잭션**(undo 로그가 작은 것)을 victim으로 선택한다.

**왜 "순서"가 문제인가**

같은 두 행을 건드리는데도, **두 트랜잭션이 락을 잡는 순서가 반대**라서 발생한다.

- T1의 순서: **A → B**
- T2의 순서: **B → A**

두 트랜잭션이 **항상 같은 순서**(예: 둘 다 `id` 오름차순 A → B)로 락을 잡았다면,
T2는 처음부터 A에서 대기했을 것이고 T1이 끝난 뒤 깔끔히 진행되어 데드락이 안 난다.
→ "자원 순서 정렬" 해결책이 효과적인 이유.

**코드 레벨에서 흔히 터지는 버전**

```java
@Transactional
public void transfer(Account from, Account to, int amount) {
    // from, to 순서가 호출자에 따라 달라짐 → 락 획득 순서가 뒤죽박죽
    lock(from);
    lock(to);
    ...
}
```

`transfer(A, B)`와 `transfer(B, A)`가 동시에 호출되면 정확히 위 데드락이 난다.
그래서 **항상 ID가 작은 계좌부터** 락을 잡도록 정렬한다.

```java
Account first  = from.getId() < to.getId() ? from : to;
Account second = from.getId() < to.getId() ? to : from;
lock(first);   // 누가 호출하든 항상 ID 오름차순
lock(second);
```

### 라이브락 — "실패하면 양보(release & retry)" 전략이 동기화될 때

```
tr1: A 락 획득 → B 락 시도 → 실패하면 A를 풀고 재시도
tr2: B 락 획득 → A 락 시도 → 실패하면 B를 풀고 재시도
```

타이밍이 완벽히 겹치면:

```
시점1: tr1 = A잡음 / tr2 = B잡음
시점2: tr1 = B실패 → A해제 / tr2 = A실패 → B해제   (동시에 양보)
시점3: tr1 = A잡음 / tr2 = B잡음                  (동시에 재시도, 원위치)
시점4: ... 무한 반복
```

각자는 "락 잡고 → 실패 감지 → 풀고 → 재시도"를 바쁘게 돌아
**CPU는 끓는데** 트랜잭션은 한 발짝도 못 나간다.

## 5. 데드락 발생의 4가지 조건 (Coffman Conditions)

아래 네 가지가 **동시에** 만족되어야 데드락이 발생한다. **하나라도 깨면 예방**된다.

1. **상호 배제 (Mutual Exclusion)** — 자원을 한 번에 하나의 프로세스만 점유 가능 (공유 불가)
2. **점유와 대기 (Hold and Wait)** — 자원을 가진 채로 다른 자원을 추가로 기다림
3. **비선점 (No Preemption)** — 다른 프로세스의 자원을 강제로 뺏을 수 없음
4. **순환 대기 (Circular Wait)** — P1이 P2의 자원을, P2가 P1의 자원을 기다리는 원형 고리

> 해결책 연결: **자원 순서 정렬**은 4번(순환 대기)을 깨고, **타임아웃/강제 종료**는 3번(비선점)을 깬다.

---

## 6. 해결책 정리

### 데드락 해결

1. **자원 순서 정렬** — 모든 트랜잭션이 항상 같은 순서(예: A → B)로만 락 획득. 순환 대기 자체를 차단 (가장 근본적인 예방책)
2. **교착 상태 감지 후 강제 종료** — DB가 데드락 감지 후 victim 트랜잭션을 롤백
3. **타임아웃 설정** — 일정 시간 후 락 획득을 포기하고 롤백

### 라이브락 해결

1. **랜덤 백오프** — 재시도 전 대기 시간에 랜덤성(jitter)을 줘서 두 트랜잭션의 주기를 엇갈리게 함
2. **재시도 횟수 제한** — N번 실패하면 포기하고 에러 처리 (무한 루프 방지)
3. **우선순위 / 타임스탬프 부여** — 더 오래된 트랜잭션이 이기고, 진 쪽만 양보 → 한쪽이 양보를 멈춰 엇갈림 종료
4. **자원 순서 정렬** — 데드락과 동일. 애초에 양보 상황이 안 생김

> **핵심 원리: 라이브락은 "대칭성을 깨는 것"으로 해결한다. 랜덤 백오프는 그 구현 중 하나일 뿐.**

## 7. 실제 시스템에서의 발생 사례

### 데드락

- DB에서 두 트랜잭션이 락 순서를 거꾸로 잡을 때 (MySQL: `Deadlock found` → 한쪽 롤백)
- Java 멀티스레드에서 두 `synchronized` / `ReentrantLock`을 역순으로 획득할 때
- Spring에서 분산락(Redisson 등) 여러 개를 일관되지 않은 순서로 잡을 때

### 라이브락

- 메시지 큐 재시도 로직에서 여러 컨슈머가 동시에 실패 → 재시도를 반복할 때 (백오프 없이)
- 분산 시스템에서 충돌 감지 후 양쪽이 동시에 롤백·재시작을 반복할 때
- `tryLock` 실패 시 락을 풀고 재시도하는 로직이 양쪽에서 동기화될 때

## 8. Java / Spring 코드 관점

```java
// 라이브락 위험: 실패하면 무조건 풀고 재시도
while (true) {
    if (lockA.tryLock()) {
        if (lockB.tryLock()) {
            try { /* 작업 */ } finally { lockB.unlock(); lockA.unlock(); }
            break;
        } else {
            lockA.unlock(); // 양보 → 상대도 동시에 양보하면 무한 반복
        }
    }
}
```

**개선 방향**

- `tryLock(timeout, TimeUnit)` 에 **랜덤 타임아웃**을 줘서 주기를 어긋나게 함
- 락을 **ID 기준으로 정렬**해 항상 같은 순서로 획득 → 데드락·라이브락 동시 예방
- DB 레벨은 보통 DB의 데드락 감지가 알아서 한쪽을 롤백해주지만,
  애플리케이션 레벨 락(`ReentrantLock`, Redisson 분산락 등)은 **직접 백오프/순서를 설계**해야 함

---

## 9. 보충: "삭제 쿼리에서도 데드락이 나는가?"

난다. `DELETE` / `UPDATE`가 잡는 **행 락 · 갭 락(gap lock)** 순서가 엇갈리면 데드락이 발생한다.
특히 MySQL InnoDB의 갭 락 때문에 **인덱스가 없는 `DELETE`** 에서 자주 발생한다.
→ 예방: 삭제 조건 컬럼에 인덱스 설정, 트랜잭션의 락 획득 순서 통일.

---

## 10. 면접용 핵심 요약

- **데드락 = blocked, CPU 한가, 서로 잡고 멈춤 / 라이브락 = running, CPU 바쁨, 서로 양보하며 헛돎**
- 데드락은 Coffman 4조건이 동시에 성립할 때 발생, 하나만 깨도 예방
- 데드락 예방의 정석은 **자원 순서 정렬**(순환 대기 차단)
- 라이브락 해결의 핵심은 **대칭성 깨기**(랜덤 백오프, 우선순위 부여 등)
- 현대 DB는 데드락을 감지해 victim을 롤백하지만, 애플리케이션 락은 직접 설계 필요
